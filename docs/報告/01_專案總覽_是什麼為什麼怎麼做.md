# 專案總覽：智慧知識庫（Knowledge Base AI）

**文件目的**：向主管完整說明本專案「是什麼、為什麼要做、怎麼做出來」，作為專案審查的第一份基礎資料。

---

## 一、是什麼（What）

**一句話定義**：一套將非結構化文件（PDF / Word / PPT / 音影片等）轉換成「知識圖譜」，並用該圖譜輔助大型語言模型（LLM）回答問題的內部知識庫系統。

### 1.1 產品形態

- 網頁應用（無框架的 Vanilla JS 前端 + FastAPI 後端），可用瀏覽器操作，也可用命令列批次處理。
- 支援兩種部署方式：Docker Compose 一鍵啟動、或本地 Python + Neo4j Desktop 開發模式。
- 使用者可建立**多個獨立知識庫（KG, Knowledge Graph）**，例如「產品文件庫」「法規知識庫」互不干擾，各自有獨立的資料與問答範圍。

### 1.2 核心能力（對應 README 特性表）

| 能力 | 說明 |
|---|---|
| 多場景知識庫 | 每個 KG 獨立管理，Neo4j Enterprise 下各自有專屬資料庫 |
| 結構化知識抽取 | 文件內容被拆解成「主詞-關係-受詞」三元組（SVO），關係限定為 30 種明確語意類型 |
| 雙層檢索問答 | 先路由到相關知識庫，再用圖遍歷找知識事實，最後回補原文佐證 |
| 自我精煉迴圈 | 答案信心不足時自動補充更多原文，最多重試 3 輪 |
| 多 LLM Provider | Ollama（本地）/ OpenAI / Anthropic / Gemini / Grok，可自由切換 |
| 自動分群建庫 | 文件先進暫存區，LLM 自動判斷該分成幾個知識庫、如何命名 |
| PDF OCR 三層備援 | pypdf → pdfminer → PaddleOCR，掃描版 PDF 也能處理 |

### 1.3 誰會用、用來做什麼

企業/團隊內部知識庫場景：把分散的文件（規格書、規章、會議紀錄、技術文件）集中管理，讓使用者用自然語言提問，得到「有出處、有邏輯依據」的答案，而不是單純關鍵字搜尋或模糊的向量相似度比對。

## 二、為什麼（Why）

### 2.1 要解決的問題：傳統 RAG 的兩個痛點

市面上常見的 RAG（Retrieval-Augmented Generation）做法是「文件切段 → 向量化 → 找語意最相似的段落 → 丟給 LLM 生成答案」。這個做法有兩個已知缺陷：

1. **只有「相似」，沒有「邏輯關係」**
   向量相似度只能告訴你「這段文字跟問題長得像」，無法表達「A 導致 B」「A 是 B 的子類別」這種明確的邏輯結構。遇到需要串多步推理的問題（例如「某個問題的根本原因牽涉到哪些上游因素」），純向量檢索容易漏掉關鍵資訊，也更容易讓 LLM 產生幻覺（answer 看似合理但實際上與原文無關）。

2. **知識圖譜三元組雖精確，但容易失去情境**
   反過來說，如果只存三元組（例如「李白 - 出生於 - 碎葉城」），雖然邏輯精確，但原文的細節（時間、數量、前後語境、例外情況）會被壓縮掉，導致回答內容單薄、缺乏依據可供追查。

### 2.2 這個專案的立場

**兩者都要，而且要能互相佐證**：用知識圖譜做「邏輯骨架」的檢索與推理，再用原文回補「血肉」的細節，並且每一步都要能追溯回原始文件段落（出處可查）。這是貫穿整個系統設計的核心動機，也是與單純「向量 RAG」或單純「Graph RAG」系統的差異所在。

### 2.3 為什麼要多知識庫、而不是單一大圖譜

單一巨大知識圖譜在文件量變大後會遇到兩個現實問題：不同領域的實體容易「撞名衝突」（例如兩個不同專案都有叫「Router」的概念），以及查詢效能隨圖規模增加而下降。因此設計上讓每個 KG 各自獨立（甚至各自一個 Neo4j 資料庫），先用「路由層」判斷問題該去哪個 KG 找答案，再進圖譜細查。

## 三、怎麼做（How）

### 3.1 系統分層架構

```
瀏覽器（Vanilla JS 前端）
        │ HTTP / SSE（串流）
        ▼
FastAPI 後端
  /agent            問答（SSE 串流）
  /knowledge-graphs  KG 管理、建圖
  /documents         文件管理、批次匯入
  /staging           暫存區、自動分群
  /search            向量搜尋
        │
        ├── LLM Provider（ollama / openai / anthropic / gemini / grok）
        ├── Embedding Provider（local / openai / ollama）
        ├── Neo4j 主資料庫（KG 元資料、路由層）
        └── Neo4j KG 專用資料庫 × N（各 KG 的知識圖譜本體）
```

### 3.2 資料怎麼進來：文件生命週期

```
原始檔案（PDF / DOCX / PPTX / MP3 / MP4）
    │ 轉譯／OCR（transcribe_file）
    ▼
暫存區 workspace/_staging/
    │ 使用者選擇「直接匯入」或「自動分群」
    ▼
KG 工作區 workspace/kg_xxx/_text/
    │ 建立知識圖譜（build_graph_for_kg）
    ▼
Neo4j：Entity 節點 + 30 種語意關係邊
```

**自動分群**是一個實用的細節：使用者不需要事先決定要建幾個知識庫，文件先丟進暫存區，由 LLM 分析內容自動建議分組與命名，使用者確認（可編輯）後才正式建立 KG 並在背景觸發知識圖譜建構。

### 3.3 文件怎麼變成知識圖譜：SVO 抽取流程

1. 文件依句子邊界切成約 1500 字的段落（chunk），並持久化保存（`chunk_store/`），之後每個知識節點都會記錄自己來自哪個 chunk。
2. 用 LLM 以 JSON 模式抽取三元組：`主詞 | 主詞類型 | 關係類型 | 動詞 | 受詞 | 受詞類型`（失敗時退回較寬鬆的 pipe 格式，並自動重試 2 次）。
3. 關係類型**限定在 30 種預先定義好的語意類型**（例如 IS_A、CAUSES、USES、PART_OF），不是讓 LLM 自由發揮的模糊分類，這樣可以用 Neo4j 的 edge label 做高效的圖形比對，也避免關係語意發散難以查詢。
4. 三元組批次寫入 Neo4j（`MERGE`），同一實體跨文件重複出現時會自動合併，並累加「信心值」。
5. 抽取失敗的段落不會標記完成，下次批次執行時會自動補跑（增量處理，不用整庫重建）。

### 3.4 使用者問問題時發生了什麼事（雙層檢索）

```
① 問題進來
   → LLM 從問題中萃取關鍵概念（最多 8 個）

② 路由層（KG 選擇）
   → 用 cosine similarity 比對問題概念與各 KG 的主題向量
   → 選出分數超過門檻的 KG（最多 5 個）

③ 知識層（圖遍歷）
   → 在選中的 KG 裡做 1-2 跳的 BFS 圖遍歷（證據稀疏時可加深到 3 跳）
   → 取出結構化知識事實：「主詞 [關係類型] 受詞」

④ 圖譜驅動的原文回補
   → 每個知識事實都記錄了來源文件與 chunk 座標
   → 依這些座標回頭抓原文段落，補上三元組壓縮掉的細節
   → 同時用向量相似度做補充檢索，避免圖譜漏掉的內容被忽略

⑤ 自我精煉迴圈
   → 若答案信心不足，重複④再多拉一些原文，最多 3 輪

⑥ LLM 生成回答
   → 知識事實 + 原文片段組成 Prompt → 串流輸出答案
```

前端會即時顯示每一步的中間結果（路由到哪些 KG、比對到哪些知識事實、精煉了幾輪），這是刻意設計的可解釋性（explainability），讓使用者知道答案「怎麼來的」，而不是黑盒生成。

### 3.5 技術棧

| 元件 | 選型 | 用途 |
|---|---|---|
| 後端框架 | FastAPI | API 服務、SSE 串流回應 |
| 圖資料庫 | Neo4j（Enterprise 多資料庫 / Community 單資料庫 fallback） | 儲存知識圖譜與路由層 |
| Embedding | sentence-transformers（本地）/ OpenAI / Ollama | 語意向量計算，本地模式不需 API 金鑰 |
| LLM | Ollama / OpenAI / Anthropic / Gemini / Grok | 三元組抽取、概念萃取、答案生成，五選一可切換 |
| OCR | PaddleOCR（+ pypdf / pdfminer 備援） | 掃描版 PDF 文字化 |
| 語音轉文字 | faster-whisper | 音影片轉譯 |
| 前端 | Vanilla JS（無框架） | 單頁應用，含深色模式 |
| 部署 | Docker Compose / Windows PowerShell 腳本 | 一鍵啟動 |

### 3.6 關鍵設計參數（可調校，非寫死邏輯）

| 參數 | 預設值 | 意義 |
|---|---|---|
| `KG_ROUTE_THRESHOLD` | 0.05 | 路由層最低分數門檻，決定要不要把某個 KG 納入查詢 |
| `MAX_KG_PER_QUERY` | 5 | 一次問答最多同時查幾個 KG |
| `SCORE_THRESHOLD` | 0.70 | 向量相似度搜尋的門檻 |
| BFS 跳數 | 1-2（最深 3） | 圖遍歷深度，依問題複雜度與證據充足度動態調整 |
| 自我精煉輪數 | 最多 3 輪，信心門檻 0.65 | 低信心答案的補救機制 |

這些參數集中在 `core/constants.py`，代表系統的檢索行為是可調校的，不是寫死在程式邏輯裡。

## 四、目前狀態的誠實補充

- 測試套件：`tests/` 下有 455 個 pytest 測試案例。
- Community 版 Neo4j（非 Enterprise）會 fallback 成單一資料庫、以 `kg_id` 屬性區隔多知識庫，功能不受影響但效能較低，這代表 Enterprise 授權（多資料庫）是效能面的建議選項而非必要條件。
- `docs/TEN_MINUTE_PITCH.md` 是一份偏學術包裝的口試簡報稿，用詞（如「神經符號」「Graph MoE」）比實際工程實作更誇飾，對主管報告時建議以本文件的說法為準，避免用詞落差造成誤解。
