2. **正式投稿前仍建議做最後一次人工確認**：本次查證是透過網路搜尋比對摘要頁與索引資料庫，不是逐篇通讀全文。對於你論文中會**重點立論**的少數幾篇（例如 GraphRAG、RAG、Self-RAG、FLARE 這幾篇如果是你論文的核心對照對象），建議親自下載 PDF 讀過摘要與方法章節，確保「理論連結」的敘述（例如本專案文件中每條引用後面的「理論連結」段落）站得住腳——查證只能確認「論文存在且標題作者正確」，無法確認「你對這篇論文的詮釋是否恰當」，這部分需要你自己判斷。

本次查核採用「多獨立查證流程、分批交叉核對」的方式，而非單一次通篇檢查——理由是先前 `THEORETICAL_ARCHITECTURE.md` 自身的稽核記錄顯示，**同一輪查證容易對同一批文獻重複同樣的盲點**（例如標題用字錯誤即使查證過 arXiv ID 是否對應正確論文，仍可能沒有逐字核對標題本身）。本次批次 C 就抓到一個前一輪稽核完全沒發現的標題錯誤（Traag et al. 的 "well-behaved" vs "well-connected"），驗證了這個方法論假設。

### 查詢方法與資料品質的誠實聲明

被引用次數本次改用 4 組獨立查詢完成，過程中 Semantic Scholar 官方 API 多次觸發速率限制（429），因此改用 **OpenAlex API**、Google Scholar 聚合搜尋、SciSpace 等來源交叉比對。這代表：

- **多數數字是合理範圍內的近似值，不是查詢當下精確到個位數的即時數字**——不同資料庫（Semantic Scholar / Google Scholar / OpenAlex）本身統計口徑就不同（例如是否把 arXiv 預印本版與正式會議/期刊版視為同一筆記錄），同一篇論文在不同資料庫可能有數倍落差。
- **少數幾筆查詢方式本身就會影響結果**：例如 Graph-CoT（arXiv:2404.07103）直接用 arXiv DOI 查詢只顯示 1 次引用，改用標題搜尋才比對到正確紀錄、變成 30 次——這個案例特別值得記錄，因為它示範了「查詢方法本身的偏誤」有多大。
- **凡標記「too new」或「UNCERTAIN」的論文，代表資料庫還來不及索引足夠引用，不代表論文品質有問題**——2025-2026 年剛發表的 arXiv 論文本來就需要數月到數年才會累積可觀引用數，這是索引延遲的正常現象。
- **正式寫進論文前，建議你對其中最核心立論的幾篇（尤其是 RAG、GraphRAG、MoE、Attention、Self-RAG、GraphSAGE 這幾篇你會重點引用的），親自到 semanticscholar.org 或 Google Scholar 手動查一次精確數字並記錄查詢日期**——本報告的數字適合用來判斷「這是奠基性論文還是邊緣論文」的量級，但不適合直接複製貼上當作論文裡的精確引用數字。
